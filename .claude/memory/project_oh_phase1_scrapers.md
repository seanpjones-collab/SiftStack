**Cuyahoga-specific learnings (2026-04-22):**
- **cpdocket TOS** explicitly prohibits URL-construction attacks and "bulk data mining without prior approval." Going through the site's own form (POST the Foreclosure Search panel) is accepted because it uses the query parser, not direct backend access. Small date-range windows (≤30 days) + human-pace delays keep the scraper out of the "permanently denied without notice" zone. Sean authorized this interpretation 2026-04-22.
- **Native date-range filter** on cpdocket (`txtFromDate`/`txtToDate` with MaskedEditBehavior). One POST per filing-type per range — much lighter than Summit clerkweb's single-date-only model.
- **Results grid has all essential fields** (defendant, address, city, zip, case #, parcel, status, filed date). No per-case detail clicks needed for MVP. Drops load on the ToS-sensitive database by ~50×.
- **DLN (Daily Legal News)** has a public WordPress REST API at `/wp-json/dln/v1/data-table?type=foreclosure-notices`. Clean ACF fields. No Playwright, no auth. Tax-foreclosure category isn't exposed on this endpoint (returns `[]`) — cpdocket filing types 1465/1466/1467 cover that exclusively.
- **Cross-source dedup is by parcel ID**, NOT case number. cpdocket uses CV-YY-NNNNNN; DLN uses internal 6-digit refs (131998, 134620). The CV number doesn't appear in DLN notice bodies either. Canonical parcel format: `NNN-NN-NNN` (zero-padded). Fallback key when parcel is missing: street-num + city-upper + owner-last-name-upper.
- **cpdocket MaskedEditBehavior** breaks both `page.fill()` and `page.keyboard.type()` (cursor-hops mid-entry). Must JS-set via native HTMLInputElement setter + input/change events. ASP.NET postback reads the form-field value directly, bypassing the mask widget.
- **TOS gate** is a one-time click-through per session at `/TOS.aspx` (btnYes). After accepting, subsequent navigation to `/Search.aspx` stays past the gate until the session cookie expires.
- **Deceased-defendant patterns in Cuyahoga** differ slightly from Summit — the Clerk uses "UNKN. HEIRS, DEVISEES OF X" and "UNKNOWN ADMINISTRATOR OF THE ESTATE OF X" as named lead defendants. Regex must cover these, not just "ESTATE OF" / "UNKNOWN HEIRS OF".

**Probate-specific learnings (2026-04-22):**
- **Summit eServices Wicket AJAX** replaces the `caseCd <select>` after every other field's onchange (date Tab, page-size change). ALWAYS select caseCd LAST or it resets to empty and the submit raises a "Case Type is required" feedback panel. Dates need `page.keyboard.type()` + Tab to fire the Wicket sync — `page.fill()` alone doesn't. `select_option(value="ES        ")` with trailing spaces silently no-ops; use `select_option(label="Estate")` instead.
- **Summit case-detail pages are Tier-1 probate data** at Day 0 — decedent name + DOD + property address + fiduciary name + fiduciary address all in structured divs. BeautifulSoup structure-based parsing (`.rowodd > .subSectionHeader2` for heading, `.ptyContactInfo > .addrLn1/2/3` + spans for address) beats regex-on-flat-text (which over-greedy-matches across street/city boundaries).
- **Summit Applicant fallback** — when fiduciary isn't yet formally appointed by the court (days 0-14 of an estate opening), the Applicant role IS the proposed PR. Fallback captures 30/30 ES cases vs 6/30 when only looking at Fiduciary rows.
- **Cuyahoga probate direct portal** (probate.cuyahogacounty.gov/pa/CaseSearch.aspx) has NO date-range filter, only case-number and party-name searches. Unusable as primary source. Pivoted to DLN `/wp-json/dln/v1/data-table?type=probate` — ~127 EST cases per 200-row page, Day-0 `date_filed` field. ACF schema for probate differs from foreclosure-notices (`proper_name` / `output` / `case_no` / `date_filed`, not web_ads/plaintiff/defendant).
- **Cuyahoga case-type filter within DLN** — prefix EST (estates), skip GRD (guardianships), MSC, WIL (will-only), ADV (adversarial), TRS. The `output` field names attorney not fiduciary, so owner_name stays empty and obituary_enricher fills via decedent search.
- **Stark probate portal is Adobe GoLive Classic ASP at `www.probate.co.stark.oh.us`** — HTTP-only (HTTPS times out). Initial "backend broken" verdict was wrong: HTTP 500 was specific to "Smith" overflowing a query limit; less-common surnames (Johnson/Williams/Brown/Jones) return clean 2-3MB result sets. Param is `case_no` on case_info.asp, NOT `txtCaseNumber` (form-name mismatch that broke initial probes).
- **Stark case-number format** — sequential integer across all case types (estate, guardianship, adoption, name change, marriage). High watermark ~255644 on 2026-04-22. Marriage cases get "M"/"MA" suffix; estates get no suffix. Case type must be filtered client-side on the `Case Type:` field of case_info.asp ("FULL ADMIN W/O WILL", "FULL ADMIN W/WILL", "RELEASE W/WILL", "RELEASE W/O WILL", "SUMMARY RELEASE*").
- **Stark ASP paired-row parsing** — case_info.asp and case_party.asp both use paired `<tr>` layouts: label row then value row. Labels render differently on the two pages: case_info uses `<font><b>Label</b></font>` (b inside font); case_party uses `<b><font>Label</font></b>` (b outside). Union regex covers both. Values uniformly use `<label><font>VALUE</font></label>`. zip(label_positions, value_positions) in document order for key→value mapping.
- **Stark APPLICANT fallback for Fiduciary** — same pattern as Summit. At Day 0 the parties table usually only has "IN REGARD TO" (decedent). Within 2-5 days the APPLICANT row lands; rare cases get a FIDUCIARY row after formal appointment. Scraper prefers FIDUCIARY, falls back to APPLICANT, emits decedent-only when neither is present yet.
- **publicnoticesohio.com is NOT needed for any of the three counties.** Direct court/DLN sources beat pnoh on latency (pnoh = newspaper = ~30-day lag) and we have Day-0 data for all three. pnoh would be a fallback if one of the direct sources disappears.
- **obituary_enricher probate-preset path** triggers when owner_name (PR) AND decedent_name are both set AND different. Summit + Stark hit this path; Cuyahoga doesn't (owner_name empty from DLN), so Cuyahoga records flow through the full obituary-search path. Both converge on the same downstream decision-maker schema.
